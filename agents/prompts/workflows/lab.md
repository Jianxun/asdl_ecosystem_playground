# Lab Workflow Prompt

This document defines the standard for designing and delivering one ASDL teaching lab.

A lab is a tightly scoped, publishable learning episode that teaches one circuit concept and demonstrates one key property or tradeoff through theory and simulation.

## Purpose

Use this prompt when creating a new lab plan or reviewing a lab draft. The output should be strong enough to become a short blog post with reproducible artifacts.

## Core principles

Every lab must satisfy these requirements:

- It is self-contained and novice-readable.
- It teaches one main concept with one explicit learning objective set.
- It includes theory-first reasoning before simulation interpretation.
- It compares theory vs simulation with explicit tolerance/margin.
- It demonstrates an important circuit property or design tradeoff.
- It is reproducible from repo root using explicit commands and paths.
- It records friction points and proposes tooling/workflow improvements.

## Canonical naming and paths

- Use the term **lab** (not experiment/exp) for teaching episodes.
- Lab id format: `lab-XX-<slug>` (example: `lab-01-rc-lowpass`).
- Source assets live under `labs/<lab-id>/`.
- Generated run artifacts live under `runs/<lab-id>/<run_id>/`.
- Keep generated figures under `labs/<lab-id>/figures/` and ensure they are script-generated.

## Required deliverables (normative)

Each lab must include all items below:

1. ASDL source files for the circuit/testbench.
2. Post-processing script(s) for simulator outputs.
3. Plot-generation script(s) for figures used in the write-up.
4. Generated figure files referenced by the write-up.
5. One markdown instruction/write-up document (`lab.md`) containing theory, setup, results, and conclusions.

Recommended folder shape:

```text
labs/<lab-id>/
  lab.md
  metadata.yaml
  asdl/
    tb.asdl
    dut.asdl                # optional
  scripts/
    postprocess.py
    plot.py
    checks.py               # optional; quantitative assertions
  figures/
    fig1_*.png
    fig2_*.png
    data/                   # optional derived analysis data
```

## `metadata.yaml` contract

Include at least:

- `id`
- `title`
- `concept`
- `difficulty`
- `learning_objectives` (list)
- `theory_checks` (list of metric/expected/tolerance)
- `simulations` (list of name/varies/demonstrates)
- `deliverables` (list of required paths)

Use this starter shape:

```yaml
id: lab-XX-<slug>
title: "<human title>"
concept: "<single main concept>"
difficulty: beginner|intermediate|advanced
learning_objectives:
  - "<objective 1>"
theory_checks:
  - metric: "<metric_name>"
    expected: "<equation or target>"
    tolerance: "<e.g., +-5%>"
simulations:
  - name: "<analysis name>"
    varies: ["<parameter>"]
    demonstrates: "<property/tradeoff>"
deliverables:
  - lab.md
  - asdl/tb.asdl
  - scripts/postprocess.py
  - scripts/plot.py
  - figures/fig1_<name>.png
```

## Required `lab.md` section order

Use this exact section order:

1. `# Lab XX: <Title>`
2. `## 1) What you will learn`
3. `## 2) Circuit overview`
4. `## 3) Theory (pre-simulation)`
5. `## 4) ASDL simulation setup`
6. `## 5) Results`
7. `## 6) Theory vs simulation`
8. `## 7) Tradeoff demonstrated`
9. `## 8) Friction log (workflow/tooling)`
10. `## 9) Reproducibility checklist`

### Section requirements

- `What you will learn`: explicit, testable learning outcomes.
- `Circuit overview`: role of the circuit, practical relevance, key parameters.
- `Theory`: assumptions, derivation, and predicted behavior.
- `ASDL simulation setup`: exact files and commands from repo root.
- `Results`: figures generated by scripts and referenced in markdown.
- `Theory vs simulation`: quantitative comparison table with error and pass criteria.
- `Tradeoff demonstrated`: clearly explain what changes when a key parameter changes.
- `Friction log`: first failing command, exact error, fix, and missing tooling notes.
- `Reproducibility checklist`: concrete checkboxes for replay confidence.

## Validation standard

Validation is mandatory. A lab is incomplete until all are true:

- Compile succeeds with explicit `asdlc` command.
- Simulation succeeds with explicit `xyce` command.
- Post-processing and plotting scripts run successfully.
- Figures in `labs/<lab-id>/figures/` match the latest run outputs.
- At least one quantitative theory-vs-simulation check meets stated tolerance.

Acceptance is observable outputs and metrics, not implementation claims.

## Formatting and writing style

- Write prose-first; use tables/lists where they improve clarity.
- Keep equations and assumptions explicit.
- Use precise, copy-runnable commands.
- Prefer concise figure captions that state what the reader should notice.
- Avoid placeholders (`TBD`) in final lab deliverables.

## Revision note requirement

When a lab is revised, update all affected sections (`metadata.yaml`, scripts, figures, `lab.md`) for consistency and append a short revision note at the bottom of `lab.md` explaining what changed and why.
