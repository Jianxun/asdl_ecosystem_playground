# Lab Pipeline Evaluation

## Pipeline Overview

```
labs/specs/<lab-id>.yaml  ──[architect]──►  labs/plans/<lab-id>.md  ──[executor]──►  labs/builds/<lab-id>/
       (stable)                                  (regenerable)                           (regenerable)
```

The spec is the stable ground truth. Plans and builds are downstream artifacts that can be
discarded and regenerated when the harness changes. This property is what makes systematic
evaluation possible: hold the spec fixed, vary the harness, compare outputs.

---

## Stage 1: Spec

**Location:** `labs/specs/<lab-id>.yaml`

The spec is the only artifact that is never regenerated. It encodes the circuit concept,
learning objective, theory checkpoint, and acceptance criteria — the authoritative definition
of what a correct lab looks like.

### Schema

```yaml
lab_id: lab-01-rc-lowpass
title: "RC Low-Pass Fundamentals"
curriculum_position: 1
difficulty: beginner

concept: "First-order RC low-pass filter"
objective: "Demonstrate cutoff frequency behavior and relate to time-domain settling."
key_tradeoff: "cutoff frequency vs settling speed"

theory:
  formula: "f_c = 1 / (2π·R·C)"
  components: {R: 100, C: 1e-6}   # ohms, farads
  expected_value: 1591.5           # Hz
  tolerance_pct: 2.0

required_analyses:
  - type: AC
    purpose: "Extract -3dB frequency"
  - type: TRAN
    purpose: "Observe settling to step input"

required_artifacts:
  - labs/builds/<lab-id>/asdl/tb.asdl
  - labs/builds/<lab-id>/scripts/plot_from_hdf5.py
  - labs/builds/<lab-id>/artifacts/tb.hdf5
  - labs/builds/<lab-id>/figures/
  - labs/builds/<lab-id>/lab.md

backend: sim.ngspice
phases: [author, compile, simulate, normalize, plot, lab-closeout]
```

### What makes a spec stable

- It describes *what* the lab must demonstrate, not *how* to implement it.
- Component values and expected values are derived from theory, not from simulation results.
- Acceptance criteria are stated before any simulation is run.
- A spec should not need to change when the harness, compiler, or simulator changes.

## Stage 2: Plan

**Location:** `labs/plans/<lab-id>.md`
**Input:** `labs/specs/<lab-id>.yaml`
**Generated by:** Architect role

The plan is the architect's translation of the spec into an executable task for the executor.
It is regenerable: if the harness improves (better priors injected, better theory framing),
a better plan can be generated from the same spec.

Note: general task ExecPlans (infra, tooling, migration tasks) live in `agents/plans/`.
`labs/plans/` is exclusively for lab pipeline plans.

### Required plan sections

- **Spec reference** — path to `labs/specs/<lab-id>.yaml`; echoes key fields (circuit_type, backend, phases)
- **Theory framing** — states formula, derivation, and expected value drawn from spec
- **Milestones** — sequenced steps with explicit exit criteria
- **Validation commands** — exact commands the executor must run with expected outcomes
- **Relevant Priors** — policies and lessons selected by querying `agents/policies/` and `agents/lessons/`
- **Acceptance criteria** — maps directly to `spec.theory` checkpoint and `spec.required_artifacts`

### Plan evaluation criteria

Evaluated independently of whether the build succeeded.

| Criterion | How to check |
|---|---|
| Theory correctly derived from spec | Compare plan formula/expected_value against spec.yaml |
| Milestones completable in one session | Flag if >5 milestones or any milestone is ambiguous |
| Validation commands are reproducible | Commands run from repo root without modification |
| Relevant priors injected | At least one policy matching `phases`; at least one lesson matching `backend` if applicable |
| Acceptance criteria are measurable | Each criterion has a numeric threshold or binary pass/fail |

### Plan failure modes

- **Theory error**: wrong formula or expected value → build will be constructed to wrong spec
- **Underspecification**: vague milestones without required outputs stated
- **Missing priors**: known failure modes not injected → executor rediscovers them
- **Scope overrun**: too many milestones for one session → incomplete build

## Stage 3: Build

**Location:** `labs/builds/<lab-id>/`
**Input:** `labs/plans/<lab-id>.md`
**Generated by:** Executor role

The build is regenerable. When the harness improves, the same plan can be re-executed
and the output compared against the previous run.

### Required build artifacts

Derived from `spec.required_artifacts`:

| Artifact | Pass condition |
|---|---|
| `labs/builds/<lab-id>/asdl/tb.asdl` | Compiles cleanly with `asdlc` |
| `labs/builds/<lab-id>/artifacts/tb.hdf5` | Exists; simulation exit code 0 |
| `labs/builds/<lab-id>/figures/` | At least one script-generated figure present |
| `labs/builds/<lab-id>/lab.md` | Contains theory section, results section, theory-sim comparison |

### Build evaluation metrics

Primary telemetry signals for measuring harness improvement.

**Efficiency metrics** (lower is better after harness improvement):

| Metric | Definition |
|---|---|
| `compile_iterations` | Tool calls to reach first clean compile |
| `simulate_iterations` | Tool calls to reach first successful simulation |
| `total_tool_calls` | Total bash/edit/write tool calls in session |
| `correction_count` | Times executor deviated from plan or self-corrected |

**Quality metrics** (pass/fail against spec):

| Metric | Definition |
|---|---|
| `theory_sim_delta_pct` | `abs(simulated - expected) / expected * 100` |
| `theory_sim_pass` | `theory_sim_delta_pct <= spec.tolerance_pct` |
| `artifacts_complete` | All `spec.required_artifacts` present at PR open |
| `known_failures_recurred` | Any injected lesson's warned failure observed in telemetry |

**Lesson effectiveness** (per injected lesson, written to `agents/episodes/`):

| Outcome | Definition |
|---|---|
| `success` | Lesson injected; warned failure did not occur |
| `failure` | Lesson injected; warned failure still occurred |

### Build failure modes

- **Known failure recurred** → lesson content needs revision; confidence score decreases
- **Compile thrash** → friction in ASDL authoring; candidate for new lesson or policy
- **Missing artifact** → milestone acceptance criteria in plan too vague
- **Theory-sim gap exceeded** → wrong circuit implementation or wrong theory in spec (distinguish by checking spec values directly)

## Evaluation Protocol

### Single run evaluation

After an executor session completes:

1. Check `artifacts_complete` against `spec.required_artifacts`.
2. Extract `theory_sim_delta_pct` from `lab.md`.
3. Record `compile_iterations` and `simulate_iterations` from session telemetry.
4. For each injected lesson, append episode outcome to `agents/episodes/<id>.jsonl`.
5. Update confidence scores in `agents/lessons/` and `agents/policies/`.

### Harness improvement rerun

To confirm a harness change improved execution:

1. Keep `labs/specs/<lab-id>.yaml` unchanged.
2. Optionally regenerate the plan from the spec (to evaluate plan improvement in isolation).
3. Re-execute in a fresh worktree: `labs/builds/<lab-id>/` is regenerable; discard and rerun.
4. Compare efficiency metrics against the prior run.
5. A confirmed improvement: at least one efficiency metric decreases and no quality metric regresses.

### Independent flow evaluation

**Evaluate plan quality without running the executor:**
- Reviewer checks plan against spec using the criteria table above.
- Record verdict in plan's reflection section.

**Evaluate build quality given a fixed plan:**
- Re-execute the same plan in a fresh worktree.
- Compare telemetry. Failures attributable to execution only when plan was rated acceptable.

---

## Improvement Attribution

| Symptom | Likely stage | Action |
|---|---|---|
| Wrong theory-sim result despite clean compile/simulate | Spec or Plan | Check spec expected_value; check plan theory framing |
| Compile thrash despite correct theory | Build / missing lesson | Add lesson for this failure mode |
| Plan missing relevant priors | Plan generation | Improve architect prompt or `query_lessons.py` trigger matching |
| Artifact missing despite successful simulation | Build / underspecified plan | Tighten milestone acceptance criteria in plan template |
| Known failure recurred despite lesson injection | Lesson content | Revise lesson or increase injection prominence |

## Artifact Location Summary

| Artifact | Location | Stable? |
|---|---|---|
| Lab spec | `labs/specs/<lab-id>.yaml` | Yes — never regenerated |
| Lab plan | `labs/plans/<lab-id>.md` | No — regenerable from spec |
| Lab build | `labs/builds/<lab-id>/` | No — regenerable from plan |
| Session telemetry | `archive/derived/events/opencode/playground/` | Append-only |
| Episode records | `agents/episodes/<id>.jsonl` | Append-only |
| Confidence scores | `agents/policies/<id>.yaml`, `agents/lessons/<id>.yaml` | Updated per episode |
