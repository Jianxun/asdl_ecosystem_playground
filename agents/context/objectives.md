# Project Objectives

Persistent goals for ASDL Playground. Future Architect sessions should read this first to keep planning aligned.

## Mission

Build a teaching-first ASDL curriculum composed of tightly scoped circuit design labs, where each lab is publishable as a short blog post and fully reproducible from repository root.

## Primary goals

1. **Teach by building labs**
   - Use the term **lab** (instead of experiment/exp) for episode-style learning units.
   - Scope each lab to one core circuit concept and one key property or tradeoff.

2. **Require theory + simulation correlation**
   - Each lab must explain the circuit and show theoretical analysis before simulation results.
   - Each lab must compare theory vs simulation with explicit approximation/error margins.

3. **Ship complete learning artifacts**
   - Each lab must include:
     - ASDL circuit sources.
     - Post-processing/analysis scripts.
     - Generated plots used as figures.
     - A markdown lab instruction document that narrates theory, setup, results, and conclusions.

4. **Use curriculum work to harden tooling**
   - Treat lab production as a workflow stress test.
   - Capture friction points and missing tooling during authoring, simulation, analysis, and reporting.
   - Convert recurring friction into updates under `agents/instructions/`, scripts, and reusable patterns.

5. **Standardize the lab data path**
   - Default simulator backend for labs is `sim.ngspice`.
   - Each lab normalizes simulator output to HDF5 using one common schema.
   - Plot generation should read from HDF5 artifacts (single input path) rather than simulator-specific CSV boilerplate.
   - Lab builds live in `labs/builds/<lab-id>/`; artifacts (simulation outputs) go to `labs/builds/<lab-id>/artifacts/` and may overwrite prior outputs by default.

## Quality bar for every lab

- The learning objective is explicit and beginner-readable.
- Simulation commands are reproducible from repo root with explicit paths.
- Result figures are generated by scripts, not hand-edited artifacts.
- At least one quantitative theory-vs-simulation check passes within stated tolerance.
- The lab demonstrates a meaningful circuit behavior, property, or design tradeoff.

## Operational implications for Architect planning

- Plan work as a sequence of small labs, not broad feature sweeps.
- Keep task metadata in GitHub Issues and store execution detail in ExecPlans.
- Ensure each lab task's deliverables include source, scripts, figures, and lab markdown.
- Require friction logging and instruction/tooling follow-ups as part of closeout.

## Non-goals

- This repo is not a full conformance suite for all circuits/backends.
- Prefer depth and clarity in a small number of high-value labs over large, shallow coverage.
